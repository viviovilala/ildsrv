# One-Click Install Design

## Overview

A single `install.sh` script that enables one-command deployment of ILDIS via:

```bash
curl -fsSL https://raw.githubusercontent.com/bphndigitalservice/ildis/main/install.sh | bash
```

Uses pre-built GHCR images. Supports both fresh installs and updates. Hosted as a raw file in the GitHub repo.

---

## Approach

**Single install.sh script** — one self-contained file handling fresh install and update. The existing `update.sh` will become a thin wrapper that calls `install.sh --update`.

---

## User Experience

### Fresh Install Flow

1. **Prerequisite checks** — Docker, Docker Compose, disk space (1GB+), port availability
2. **Interactive wizard** — prompts for configuration
3. **Setup** — creates install directory, generates `.env` and `docker-compose.yml`
4. **Pull & start** — `docker compose pull && docker compose up -d`
5. **Wait & verify** — polls health endpoint, runs migrations
6. **Print success** — URL, version, next steps

### Update Flow (existing install detected)

1. Detects `.env` and `docker-compose.yml` in directory
2. DB backup
3. Pull new images, restart containers
4. Run migrations
5. Health check
6. Print update summary

| Prompt | Variable | Default | Notes |
|--------|----------|---------|-------|
| Install directory | `INSTALL_DIR` | `/opt/ildis` | Where compose files live |
| App port | `PORT` | `8080` | Host port mapping |
| Public domain | `PUBLIC_DOMAIN` | `http://localhost:8080` | Used for feed URL generation |
| Database type | `DB_TYPE` | `mariadb` | Choices: `mariadb`, `mysql`, `external` |

| DB port (external) | `DB_DATABASE_PORT` | `3306` | |
| DB user (external) | `DB_USER` | — | Required if external |
| DB password (external) | `DB_PASSWORD` | — | Required if external |
| DB name (external) | `DB_DATABASE` | `ildis_v4` | |

## Database Configuration

### Built-in MariaDB (default)

- Image: `mariadb:10.11`
- Included as a service in docker-compose.yml
- Root password set from `DB_PASSWORD` env var
- Health check with `mysqladmin ping`
- Volume: `mysql_data:/var/lib/mysql`

### Built-in MySQL

- Image: `mysql:8.0`
- Same structure as current docker-compose.yml
- Selected when user chooses `mysql` in wizard

### External Database

- No database container in docker-compose.yml
- `app` service has no `depends_on` for MySQL
- Install script tests connectivity before starting the app
- `DB_HOST` set to user-provided value instead of service name

## Generated `docker-compose.yml`

References GHCR images instead of building from source:

- `ghcr.io/bphndigitalservice/ildis:latest` — app service
- `ghcr.io/bphndigitalservice/ildis-cron:latest` — cron service
- `mariadb:10.11` or `mysql:8.0` — database service (if not external)
- Always includes the cron service with `feed_data` shared volume

No separate `docker-compose.cron.yml` — production compose always includes cron.

## Script Structure

```
install.sh
├── Constants (GHCR repo, version, default port)
├── Helper functions (info/success/warn/fail, generate_random_key, wait_for_healthy)
├── detect_existing_install()
├── prerequisites() — Docker, disk, port checks
├── interactive_wizard() — prompts for config
├── generate_env() — writes .env
├── generate_compose() — writes docker-compose.yml
├── do_install() — pull, up, wait, migrate
├── do_update() — backup, pull, restart, migrate, health check
└── main() — detect mode, dispatch
```

`update.sh` becomes a thin wrapper: `exec bash install.sh --update "$@"`

## Non-Interactive Mode

For CI/automation:

```bash
install.sh --non-interactive
```

Reads all values from env vars or an existing `.env` file. Skips all prompts.

## Error Handling

### Pre-flight failures (exit immediately)
- Docker not installed → print install instructions
- Docker Compose not available → print install instructions
- Insufficient disk space → show available vs needed
- Port conflict → suggest alternative or flag to change

### Install failures
- `docker compose pull` fails → network/proxy message, exit
- `docker compose up -d` fails → suggest `docker compose logs`, exit
- Health check timeout (60s) → likely causes, suggest `docker compose logs app`

### Update failures
- DB backup fails → abort, print recovery steps
- Migration fails → warn but continue
- Health check fails → print backup path and rollback instructions

### Idempotency
- Re-running in same directory → detect existing install, offer update mode
- Containers already running → ask restart or abort
- `.env` exists but containers down → offer to start
